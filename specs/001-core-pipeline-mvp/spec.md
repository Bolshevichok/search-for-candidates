## Clarifications

### Session 2026-07-10

- Q: FR-002 говорит «не пересчитывая fuzzy-сравнение на каждый прогон» — это про кэш в рамках
  одного `run`, или про персистентный кэш между прогонами (тогда нужна отдельная таблица
  `departments`)? → A: Только в рамках одного `run` (in-memory кэш); персистентный кэш между
  прогонами не требуется в этой фиче — детерминированность резолва обеспечивается входными
  данными (`/sveden/struct` + текст подразделения), а не хранимым состоянием, так что повторный
  резолв на следующем прогоне безопасен и не считается «пересчётом» в смысле этого требования.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Собрать сотрудников вузов из слоя 1 (Priority: P1)

Оператор запускает сбор данных с сайтов вузов (`/sveden/employees`) по реестру и получает выгруженный,
дедуплицированный список сотрудников по каждому вузу — без этого шага у остальных шагов нет данных для работы.

**Why this priority**: Это фундамент — без данных слоя 1 нет ни матчинга, ни экспорта, ни задела для будущих
слоя 2 / VK. Это первое, что должно заработать надёжно.

**Independent Test**: Прогнать `layer1` (шаг или полный `run`) на реестре из 5 вузов и убедиться, что
`candidates`/`employees_raw` содержат по одной записи на реального человека (повторы по разным
образовательным программам схлопнуты), а недоступные вузы попали в `university_errors`, не прервав остальной
прогон.

**Acceptance Scenarios**:

1. **Given** реестр из 5 вузов с доступными страницами `/sveden/employees`, **When** оператор запускает шаг
   layer1, **Then** сырые записи сотрудников сохраняются по каждому вузу, а повторные записи одного и того же
   человека (разные образовательные программы) схлопываются в одну запись с объединёнными `disciplines`.
2. **Given** один из 5 вузов реестра, чья `/sveden/employees` после повторов всё равно отдаёт 5xx/таймаут,
   **When** выполняется шаг layer1, **Then** этот вуз фиксируется в `university_errors`, а обход остальных 4
   вузов продолжается без остановки прогона.
3. **Given** прогон, прерванный после обработки 3 из 5 вузов (например, крах процесса), **When** оператор
   перезапускает `run`, **Then** пайплайн продолжает с 4-го вуза, не повторяя обход первых трёх.

---

### User Story 2 - Обогатить карточки данными ВАК и определить статус совпадения (Priority: P2)

Оператор запускает выгрузку ВАК и матчер, чтобы к данным слоя 1 добавились защиты/степени из ВАК, а каждая
карточка получила один из 4 статусов совпадения — включая сохранение карточек, которые есть только в одном
из источников.

**Why this priority**: Это то, что превращает «список сотрудников сайта» в «карточку кандидата» — собственно
задача этой ветки («слои 1, вак, метчер»). Зависит от данных User Story 1.

**Independent Test**: При уже собранных данных слоя 1 выполнить шаги `vak` и `match` и убедиться, что у
каждой итоговой карточки ровно один из 4 статусов, что запись только из ВАК не потерялась (`vak_no_site`), а
пара с противоречащими сигналами не превращается в 5-й статус, а создаёт связь `possible_namesakes` с
`needs_review = true` на обеих карточках без исключения их из дальнейшей обработки.

**Acceptance Scenarios**:

1. **Given** запись ВАК и запись сайта на одного человека в одном (с учётом алиасов) вузе с согласующейся
   степенью, **When** выполняется матч, **Then** итоговая карточка получает `match_status = site_and_vak`.
2. **Given** запись ВАК, чьё ФИО не находится ни на одном сайте вуза из реестра, **When** выполняется матч,
   **Then** создаётся карточка `vak_no_site` (запись не отбрасывается).
3. **Given** запись сайта и запись ВАК с одинаковым ФИО, но противоречащими сигналами (степень/специальность),
   **When** выполняется матч, **Then** обе записи сохраняют свои обычные статусы (`site_no_vak` /
   `vak_no_site`), создаётся связь в `possible_namesakes`, и на обеих карточках выставляется
   `needs_review = true`.
4. **Given** вуз с флагом `is_pilot = true`, **When** выполняется шаг vak для этого вуза, **Then** запрашиваются
   записи из обеих веток API (`is_pilot=false` и `is_pilot=true`).

### User Story 3 - Один прогон без слоя 2 и VK (Priority: P3)

Оператор запускает единую команду `run` при выключенных `layer2`/`vk` и получает готовый xlsx-файл —
основной пользовательский сценарий этой ветки: полный пайплайн работает без двух шагов, которые пока
не реализуются.

**Why this priority**: Это то, что связывает User Story 1 и 2 в единый прогон и подтверждает минимальное
требование задачи — прогнать без слоя 2 и VK. Приоритет ниже первых двух, так как зависит от их данных.

**Independent Test**: С конфигом `layer2: false`, `vk: false` запустить `python -m app run --config
config.yaml` один раз и убедиться, что получен один xlsx-файл с листами `candidates`, `possible_namesakes`,
`university_errors`, `run_meta`, без ошибок про отсутствующие слой 2 / VK и без единого интерактивного шага.

**Acceptance Scenarios**:

1. **Given** конфиг по умолчанию (`layer2: false`, `vk: false`), **When** оператор запускает `run`, **Then**
   ingest выполняет layer1 и vak **параллельно** (до match), затем match → export; прогон завершается одним xlsx,
   не вызывая код слоя 2 или VK.
2. **Given** оператор всё же включил `layer2: true` или `vk: true` в конфиге до того, как эти шаги
   реализованы, **When** запускается `run`, **Then** пайплайн сразу завершается понятной ошибкой «шаг не
   реализован» вместо того, чтобы молча пропустить флаг или отдать частично неверный экспорт.
3. **Given** уже успешно завершённый `run`, **When** оператор повторно вызывает `export` без повторного
   `run`, **Then** формируется эквивалентный xlsx из уже собранных данных `data/state.sqlite`, без новых
   сетевых запросов.

---

### Edge Cases

- Что происходит, если домен вуза из реестра недоступен целиком (DNS/сеть), а не только `/sveden/employees`
  вернул ошибку? → тот же путь, что и любая другая недоступность: запись в `university_errors`, прогон
  остальных вузов продолжается.
- Что происходит, если ФИО одного человека отличается между сайтом и ВАК не из-за формата (регистр, `ё`,
  пробелы), а по существу (опечатка, добавлено отчество)? → вне зоны ответственности матчера на этом этапе:
  сравнение строгое посимвольное после нормализации, без fuzzy по ФИО — такие записи остаются
  несматченными (`site_no_vak` / `vak_no_site`), это ожидаемое поведение, не баг.
- Что происходит при самом первом запуске, когда `data/state.sqlite` ещё не существует? → файл создаётся
  заново, первый `run` всегда полный (без пропуска «уже собранного»).
- Что происходит при `app reset`? → перед сбросом состояния автоматически делается бэкап
  `data/state.sqlite` в `data/backups/`.
- Что происходит, если у вуза в реестре указан `vk_group_id`, но модуль VK ещё не реализован? → значение
  просто хранится и не используется; ошибки не возникает, так как шаг VK в этой фиче не вызывается вообще.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА разбирать страницу `/sveden/employees` каждого вуза из реестра по стандартной
  `itemprop`-микроразметке (Приказ №1493) и сохранять сырые записи сотрудников (ФИО, должность, степень,
  подразделение, `department_id`, стаж `genExperience`/`specExperience`, `source_url`) с привязкой к вузу.
- **FR-002**: Система ДОЛЖНА резолвить канонический `department_id` для каждой записи сотрудника через
  fuzzy-сопоставление названия подразделения со списком из `/sveden/struct`, сохранять результат один раз в
  рамках текущего прогона и переиспользовать его везде ниже по пайплайну этого прогона, не повторяя
  fuzzy-сравнение дважды для одного и того же подразделения внутри одного `run` (см. Clarifications). Кэш
  не обязан быть персистентным между прогонами: следующий `run` резолвит `department_id` заново из
  `/sveden/struct`, что безопасно, так как результат детерминирован входными данными.
- **FR-003**: Система ДОЛЖНА дедуплицировать записи сотрудников внутри одного вуза по `identity_key`
  (нормализованное ФИО + вуз + `department_id` + стаж, с допуском ±1–2 года между прогонами), объединяя
  дубли по разным образовательным программам в одну карточку с объединением (union) `disciplines` без потери
  и без дублирования записей.
- **FR-004**: Система ДОЛЖНА выгружать объявления о защитах из API ВАК: list `/api/att/adverts/` (пагинация,
  чекпоинт) и **detail** `/api/att/adverts/{id}/` на каждую запись (specialty, branch, defend_org и др.);
  для вузов с флагом `is_pilot` запрашивать записи из ОБЕИХ веток API (`is_pilot=false` и `is_pilot=true`).
- **FR-005**: Система ДОЛЖНА нормализовывать ФИО (регистр, `ё → е`, пробелы, дефисы в двойных фамилиях) и
  сопоставлять организацию ВАК с вузом реестра через словарь официальных названий/алиасов перед сравнением.
- **FR-006**: Система ДОЛЖНА присваивать каждой итоговой карточке кандидата ровно один из 4 статусов:
  `site_and_vak`, `site_and_vak_probable`, `vak_no_site`, `site_no_vak` — без исключений и без 5-го значения.
- **FR-007**: Если ФИО совпало, но сигналы противоречат (степень сайта ниже, чем предполагает запись ВАК,
  или специальность/дисциплины из несвязанных областей), система ДОЛЖНА НЕ мержить записи и не вводить 5-й
  статус, а создать связь в отдельной таблице `possible_namesakes` и выставить `needs_review = true` на обеих
  карточках, не меняя их обычный `match_status` и не исключая их из дальнейшей обработки.
- **FR-008**: Система ДОЛЖНА сохранять сырые и производные данные в SQLite (`data/state.sqlite`) с
  чекпоинтами по каждому шагу (по вузу — для layer1, по странице/курсору — для ВАК), чтобы прерванный `run`
  продолжался с последнего завершённого чекпоинта, а не с нуля.
- **FR-009**: Система ДОЛЖНА применять повтор с экспоненциальным backoff (до 3 попыток, 1с → 2с → 4с) на
  сетевые ошибки и 5xx как для API ВАК, так и для запросов к сайтам вузов, через единую политику повторов
  (одно место в коде, не отдельная логика на каждый источник), и НЕ ДОЛЖНА повторять запросы при
  4xx-ошибках.
- **FR-010**: Система ДОЛЖНА предоставлять CLI-команды `run`, `step layer1`, `step vak`, `step match`,
  `export`, `status`, `reset`, где `run` выполняет ingest (`layer1` ∥ `vak` через `app/pipeline/ingest.py`,
  если оба включены) → `match` → `export` согласно флагам в `config.yaml`.
- **FR-011**: Система ДОЛЖНА поддерживать полный прогон при `layer2: false` и `vk: false` в `config.yaml`
  (значение по умолчанию для этой фичи) и при этом выдавать полный, валидный xlsx-экспорт — то есть
  `run`/`export` НЕ ДОЛЖНЫ жёстко зависеть от layer2/VK как обязательных шагов в рамках этой фичи.
- **FR-012**: Система ДОЛЖНА экспортировать `output/candidates_<date>.xlsx` с листами `site_employees`
  (сотрудники вузов: post, academic_title, gen/spec_experience, source_url и др.), `vak_candidates`
  (записи ВАК с полями detail API), `possible_namesakes`, `university_errors`, `run_meta`. Слияния
  (`site_and_vak` / `site_and_vak_probable`) — на обоих листах людей. Колонки layer2/VK присутствуют на
  `site_employees`, но пустые в этой фиче.
- **FR-013**: Система ДОЛЖНА фиксировать вузы, у которых `/sveden/employees` не открылась/не распарсилась,
  в листе `university_errors`, а не молча пропускать их из прогона.
- **FR-014**: Система ДОЛЖНА автоматически бэкапить `data/state.sqlite` в `data/backups/` после каждого
  успешного `run` и перед каждым `reset`, храня ограниченное число последних бэкапов.
- **FR-015**: Система ДОЛЖНА содержать пустые заготовки для будущих модулей layer2 (краулер + ИИ-парсер
  контактов) и VK-интеграции — с описанием их будущего контракта входа/выхода (см.
  `candidate-pipeline-architecture.md`, §6.6 и §7), но НЕ ДОЛЖНА реализовывать в них саму логику краулинга,
  ИИ-парсинга или вызовов VK API; ни одна CLI-команда не должна вызывать их в рамках этой фичи.
- **FR-016**: Если оператор всё же выставит `layer2: true` или `vk: true` в конфиге до реализации этих
  шагов, система ДОЛЖНА немедленно завершиться понятной ошибкой «шаг не реализован», а не молча
  игнорировать флаг или выполнять частичный/обманчивый экспорт.
- **FR-017**: Система ДОЛЖНА вычислять и сохранять поля `candidate_content_hash` (на карточку) и
  `university_site_hash` (на вуз, в `runs`/`run_steps`) на каждом прогоне, чтобы будущая фича могла на их
  основе реализовать инкрементальные прогоны без изменения схемы данных — сам пропуск повторной обработки
  по этим хэшам в рамках этой фичи НЕ реализуется (см. Assumptions).

### Key Entities

- **University** (`universities`): вуз из `university_registry` — `official_name`, `aliases`, `domain`,
  `region`, `accreditation_status`, `vk_group_id`, `is_pilot`, `layer1_status`.
- **EmployeeRaw** (`employees_raw`): сырая запись сотрудника с сайта вуза — ФИО, должность, степень,
  подразделение, `department_id`, стаж, `source_url`, привязка к вузу.
- **VakRecord** (`vak_raw`): сырая запись объявления/защиты из ВАК — ФИО, организация защиты, тип
  диссертации, специальность, отрасль, дата, `is_pilot`-ветка происхождения.
- **Candidate** (`candidates`): единая карточка кандидата — `candidate_id`, `full_name`, `identity_key`,
  `match_status` (закрытый список из 4), `needs_review`, `university`/`department`/`degree`/`disciplines`,
  массив `defenses[]`, `candidate_content_hash`, `first_seen_run_id`/`last_seen_run_id`.
- **PossibleNamesake** (`possible_namesakes`): связь между карточкой сайта и карточкой ВАК с одинаковым ФИО
  и противоречащими сигналами — `site_candidate_id`, `vak_candidate_id`, `reason`.
- **Run / RunStep** (`runs`/`run_steps`): метаданные прогона — что сделано, ошибки по вузу/шагу,
  `university_site_hash` по вузу для будущей инкрементальности.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: После одного запуска пайплайна на реестре из N вузов получается один xlsx-файл с одной строкой
  в `candidates` на каждого отдельного человека, найденного на `/sveden/employees` любого вуза реестра и/или
  сматченного с ВАК — без ручного шага оператора между запуском и получением файла.
- **SC-002**: Если прогон прерван после обработки K из N вузов и перезапущен, он успешно завершается без
  повторного скачивания данных по первым K вузам.
- **SC-003**: 100% строк листа `candidates` имеют ровно одно из 4 утверждённых значений `match_status`; 0
  строк с неопределённым/пятым статусом.
- **SC-004**: Каждая карточка, упомянутая в `possible_namesakes`, при этом полностью присутствует в
  `candidates` со всеми доступными полями слоя 1/ВАК — то есть она не исчезла из основной выгрузки из-за
  флага `needs_review`.
- **SC-005**: xlsx-экспорт успешно формируется и открывается в Excel даже когда `layer2` и `vk` выключены в
  конфиге — колонки контактов/VK присутствуют (чтобы в будущем включение этих шагов не требовало менять
  структуру файла), но остаются пустыми.
- **SC-006**: Оператор может запустить пайплайн на реестре из сотен вузов без единого запроса
  подтверждения/интерактивного ввода и получить единственный xlsx-файл как итоговый артефакт.
- **SC-007**: Повторный вызов `export` (без повторного `run`) после успешного прогона воспроизводит
  эквивалентный xlsx из уже собранных данных без каких-либо новых сетевых запросов.

## Assumptions

- `data/university_registry.csv` уже существует и заполнен (см. `university_registry/README.md`); эта фича
  только читает реестр, не перестраивает и не валидирует его дополнительно сверх уже существующих скриптов
  в `university_registry/`.
- Доступ к API ВАК не требует токена/авторизации (по данным `research/BAK/vak-analysis.md`) — переиспользуем
  уже проведённое исследование, отдельного шага на этой фиче нет.
- Инкрементальные прогоны (пропуск неизменившихся вузов/карточек по хэшам) осознанно откладываются на
  отдельную будущую фичу — в рамках этой фичи каждый `run` полностью пересобирает layer1/ВАК/матч, но поля
  `candidate_content_hash`/`university_site_hash` уже считаются и сохраняются, чтобы не менять схему позже.
- Headless-браузер и пул прокси для заблокированных/JS-рендерящихся сайтов вузов откладываются на будущую
  фичу; в рамках этой фичи слой 1 работает только через обычные HTTP-запросы (`httpx` + `lxml`/selectolax),
  а неудачи логируются в `university_errors` для ручного разбора.
- Модули `layer2` и `vk` — пустые заготовки (папки/файлы с докстрингом контракта), не рабочий код; их
