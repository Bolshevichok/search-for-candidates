# Research: Core Pipeline Foundation (Layer 1 + VAK + Matcher)

Все технические решения ниже — прямое применение уже проведённого исследования в `research/`, не
новые изыскания. Формат: Decision / Rationale / Alternatives considered.

## 1. Стратегия тестирования

**Decision**: `pytest`, только unit-тесты на чистую логику без сети (нормализация, `identity_key`,
матчер, хэши, форма xlsx-строки). Никаких contract/integration-тестов, которые реально ходят в сеть
на ВАК API или сайты вузов. Сквозная проверка полного `run` — вручную по `quickstart.md`, не
автотестом в CI.

**Rationale**: Конституция сознательно не фиксирует стратегию тестирования проекта (см. Sync Impact
Report в `constitution.md` — «deferred»), но само наличие тестов на код с нетривиальной логикой
(матчер, дедуп, нормализация ФИО) — не противоречит Принципу I (Simplicity First): это не
абстракция сверх задачи, а самый дешёвый способ не сломать эти правила при рефакторинге. Тесты,
которые дёргают реальный ВАК API или реальный сайт вуза, недетерминированные (сервер медленный,
структура сайта меняется, `is_pilot`-ветки живые) — гонять их в CI бессмысленно и дорого; надёжность
внешних вызовов и так покрыта retry/backoff (FR-009) и `university_errors` (FR-013).

**Alternatives considered**: полноценные integration-тесты с записанными HTTP-фикстурами (VCR-style)
— отложено: усложняет фичу без прямого запроса от задачи; можно добавить отдельной фичей, если
матчер/парсер начнёт регрессировать незаметно.

## 2. CLI-библиотека

**Decision**: стандартный `argparse` с подпарсерами (`run`, `step layer1|vak|match`, `export`,
`status`, `reset`).

**Rationale**: `argparse` — часть stdlib, ноль новых зависимостей для самой частой части кода
(точка входа). Принцип I конституции: «не тащить абстракцию сверх задачи». Набор команд фиксированный
и небольшой (5 верхнеуровневых, 3 под `step`) — не тот масштаб CLI, где `typer`/`click` реально
экономят код.

**Alternatives considered**: `typer` (упомянут как опция в `app-architecture.md`, §3) — даёт красивее
`--help` и валидацию типов из аннотаций, но тянет `click` как транзитивную зависимость. Отклонено:
выигрыш не оправдывает лишнюю зависимость для 8 команд без сложной валидации аргументов.

## 3. Fuzzy-сравнение названий (вуз-алиасы, `department_id`)

**Decision**: `rapidfuzz`.

**Rationale**: Нужно в двух местах — сопоставление `defend_org` (ВАК) с `official_name`/`aliases`
вуза (§4.1 `candidate-pipeline-architecture.md`) и сопоставление `department` из слоя 1 с
`/sveden/struct` для получения канонического `department_id` (§6.2, шаг 2, там же). `rapidfuzz` —
переписанный на C++ форк `fuzzywuzzy`, MIT-лицензия (у `fuzzywuzzy`/`python-Levenshtein` — GPL,
что для чужого потребителя кода — риск), быстрее на больших списках алиасов/подразделений.

**Alternatives considered**: `fuzzywuzzy` — отклонено из-за GPL-зависимости (`python-Levenshtein`) и
более низкой скорости; ручная реализация расстояния Левенштейна — отклонено, не задача этой фичи
писать алгоритмы, которые есть готовыми и проверенными (Принцип I — не изобретать сверх задачи).

## 4. Слой 1: откуда реально брать данные (не одна страница)

**Decision**: `/sveden/employees` — это индексная страница со списком образовательных программ, у
каждой — ссылка на отдельную страницу с `itemprop="teachingStaff"` блоками. Layer1 должен: (1)
скачать индекс, (2) собрать все ссылки на страницы программ, (3) скачать и распарсить каждую по
`itemprop`, (4) дедуплицировать по `identity_key` (один человек часто в нескольких программах).

**Rationale**: Подтверждено вживую на ТюмГУ и СПбГУ (`university-sites-analysis.md`, §2). Если
считать `/sveden/employees` листом с данными напрямую — ничего не найдётся, это не соответствует
реальной структуре сайта.

**Alternatives considered**: нет — это не альтернативный выбор, а факт устройства источника,
подтверждённый исследованием.

## 5. ВАК API: конкретные вызовы

**Decision**: прямой JSON API `https://vak.gisnauka.ru`, без HTML-скрейпинга:
- Листинг: `GET /api/att/adverts/?page=N&pageSize=100&is_pilot=false` (и отдельно `is_pilot=true`
  для вузов с `is_pilot=true` из реестра — обе ветки, не одна, см. FR-004);
- Полная карточка (если понадобится точечно): `GET /api/att/adverts/<id>/`;
- Таймаут 8–10 сек/запрос (сервер отвечает 3–5 сек — это нормальный режим, не сбой), retry по
  политике §Constraints в plan.md.

**Rationale**: Проверено вживую (`vak-analysis.md`, §2) — API открыт, без авторизации, без CORS и
капчи, отдаёт чистый JSON. Поля ответа (`fio`, `dissertation_type`, `specialty`, `branch`,
`dissertation_name`→`topic`, `defend_org`, `date_defend`, `old_id`/`id`) напрямую покрывают
`defenses[]` из единой карточки кандидата (`candidate-pipeline-architecture.md`, §5) — маппинг
прямой, без промежуточной трансформации схемы.

**Alternatives considered**: HTML-парсинг публичного сайта ВАК — отклонено, JSON API строго проще
и официально задокументирован в исследовании как рабочий.

## 6. Инкрементальность в рамках этой фичи

**Decision**: не реализуем skip-логику по хэшам в этой фиче (см. Assumptions в `spec.md`), но
считаем и сохраняем `candidate_content_hash`/`university_site_hash` с первого прогона.

**Rationale**: Явное продуктовое решение задачи этой ветки — «база: слои 1, вак, метчер», без
доп. требования к инкрементальности. Считать хэши сейчас дешевле, чем добавлять их в схему поздним
патчем, когда уже будут реальные строки в базе (Принцип III — идемпотентность/чекпоинты не про
инкрементальность контента, а про резюмируемость прогона, что FR-008 уже покрывает отдельно).

**Alternatives considered**: полноценная skip-логика сразу — отклонено, за рамками задачи ветки,
кандидат на отдельную будущую фичу.
