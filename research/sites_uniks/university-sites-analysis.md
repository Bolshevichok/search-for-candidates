# Источник: сайты университетов

Что можно получить по кандидату с сайтов вузов, почему это самый предсказуемый источник из всех и как это технически устроено на уровне закона, а не только вёрстки.

**Ключевые цифры:**
- Обязательный раздел с преподавателями по строго заданному адресу — **есть у каждого аккредитованного вуза**
- E-mail / телефон в этом обязательном разделе — **не гарантированы законом**, зависят от вуза
- Стандартизированная микроразметка данных (не просто HTML-таблица «на глаз») — **есть, но соблюдение нужно проверять по каждому вузу**

## 1. Почему это надёжный источник: закон, а не добрая воля вуза

Каждая аккредитованная образовательная организация обязана вести раздел «Сведения об образовательной организации» на своём сайте — это требование **Приказа Рособрнадзора № 1493 от 04.08.2023** (в редакции Приказа № 1353 от 03.07.2025). Раздел состоит из 14 фиксированных подразделов, и что важно — **у каждого подраздела законом закреплён конкретный адрес страницы**, одинаковый для всех вузов:

| Подраздел | Адрес страницы |
|---|---|
| Основные сведения | `<домен>/sveden/common` |
| Структура и органы управления | `<домен>/sveden/struct` |
| Документы | `<домен>/sveden/document` |
| Образование | `<домен>/sveden/education` |
| Образовательные стандарты | `<домен>/sveden/eduStandarts` |
| Руководство | `<домен>/sveden/managers` |
| **Педагогический состав** | **`<домен>/sveden/employees`** |
| Матобеспечение и доступная среда | `<домен>/sveden/objects` |
| Международное сотрудничество | `<домен>/sveden/inter` |

Т.е. чтобы найти преподавателей практически любого российского вуза, не нужно даже искать раздел через навигацию сайта — можно сразу идти на `<домен вуза>/sveden/employees`. Это проверено вживую: у ТюмГУ (`utmn.ru/sveden/employees`) и у СПбГУ (`spbu.ru/sveden/employees`) страница по этому адресу реально существует и оформлена по стандарту.

## 2. Структура страницы `/sveden/employees` (проверено на живых сайтах)

Страница — это **не список людей напрямую**, а список образовательных программ вуза, у каждой из которых есть ссылка («Перейти») на отдельную страницу со списком преподавателей именно этой программы:

```
/sveden/employees  (индексная страница)
 ├── Программа 1 → страница со списком ППС этой программы
 ├── Программа 2 → страница со списком ППС этой программы
 └── ... (десятки-сотни программ у крупного вуза)
```

Это подтверждено на ТюмГУ (список из десятков программ СПО/бакалавриата/магистратуры с колонкой «Ссылка на страницу с информацией о ППС») и на СПбГУ (программы сгруппированы по уровням образования и направлениям, у каждой — свой код и ссылка). Один и тот же преподаватель обычно фигурирует в нескольких программах — это создаёт дублирование, которое нужно учитывать при парсинге (дедуплицировать по ФИО/странице сотрудника).

## 3. Единая микроразметка данных — можно парсить не «на глаз», а по атрибутам

Это ключевая находка, которой не было в предыдущем срезе исследования. Приказ №1493 требует не просто показать данные текстом, а **обернуть их в стандартные HTML-атрибуты `itemprop`**, одинаковые для всех вузов страны:

| Поле | Атрибут `itemprop` |
|---|---|
| Блок преподавателя целиком | `teachingStaff` |
| ФИО | `fio` |
| Должность | `post` |
| Преподаваемые дисциплины | `teachingDiscipline` |
| Уровень образования / специальность / квалификация | `teachingLevel` |
| Направление подготовки | `employeeQualification` |
| Учёная степень | `degree` |
| Учёное звание | `academStat` |
| Повышение квалификации / переподготовка | `profDevelopment` |
| Общий стаж работы | `genExperience` |
| Стаж по специальности | `specExperience` |
| Образовательная программа | `teachingOp` |

**Практическое следствие:** теоретически можно написать один универсальный парсер, который ищет `itemprop="teachingStaff"` и внутри него читает `fio`, `degree`, `teachingDiscipline` и т.д. — вместо того чтобы писать отдельный парсер под вёрстку каждого вуза. Это может кардинально упростить масштабирование на много вузов.

**Но: это требует обязательной ручной перепроверки.** Приказ обязывает вузы использовать эту разметку, но соблюдение — вопрос качества исполнения на стороне конкретного вуза, а не гарантия. На практике стоит:
1. Выбрать выборку из 5–10 вузов, реально интересных для найма (не только ТюмГУ/СПбГУ), и проверить через просмотр исходного кода страницы, действительно ли там присутствуют атрибуты `itemprop`, а не просто визуально похожая таблица без разметки.
2. Проверить, не блокирует ли `robots.txt` конкретного вуза путь `/sveden/` (маловероятно, так как это законодательно обязательный публичный раздел, но встречались вузы с кастомными CMS и нестандартным поведением).
3. Проверить, как ведут себя вузы с очень старыми/самописными сайтами — там, где обязаловка выполнена формально (просто вставили таблицу, но без itemprop) — на такой случай нужен fallback-парсер по обычной структуре таблицы.

## 4. Чего в обязательном разделе НЕТ — и где это искать вместо него

Приказ не требует публикации личных контактов (e-mail, телефон) в разделе «Педагогический состав» — это на усмотрение вуза. По прошлому срезу исследования (карточки сотрудников ТюмГУ и СПбГУ) контакты чаще находятся не на индивидуальной странице сотрудника, а в **табличных списках кафедр** (там, где перечислены все сотрудники кафедры разом, часто с общими или предсказуемыми по шаблону e-mail):

- `и.о.фамилия@utmn.ru` — ТюмГУ, например `n.v.bagapova@utmn.ru`
- `и.фамилия@spbu.ru` — СПбГУ, например `s.shanygin@spbu.ru`

Даже если на конкретной странице e-mail не показан, шаблон вуза можно вывести из уже найденных адресов коллег по этой же кафедре и проверить гипотезу.

## 5. Свод по полям из таблицы кандидата

| Поле | Раздел «Педагогический состав» (обязательный, по `itemprop`) | Страница кафедры / индивид. страница |
|---|---|---|
| ФИО | да (`fio`) | да |
| Город | нет (только домен/адрес вуза) | город вуза |
| Возраст / дата рождения | нет | иногда: год рождения на сайте диссовета (УрФУ), год окончания / стаж — на индивид. странице |
| Степень | да (`degree`) | не всегда |
| Дата защиты степени | нет | да — на сайте диссовета вуза (`dissovet.<домен>`); реже на индивид. странице |
| Университет / кафедра | косвенно (сама структура сайта) | да |
| Дисциплины | да (`teachingDiscipline`) | не всегда |

| E-mail | нет | иногда, + шаблон вуза |
| Телефон | нет | редко личный |

## 6. Что это значит для проекта

Раздел `/sveden/employees` — самый надёжный и предсказуемый по структуре источник для охвата (найти *кого вообще* можно рассматривать: ФИО, степень, звание, дисциплины, стаж), но не для контактов. Для контактов нужен второй проход — по страницам кафедр того же вуза, где чаще встречаются e-mail или хотя бы шаблон для его вычисления.

Прежде чем закладывать это в архитектуру: нужна ручная выборочная проверка соблюдения `itemprop`-разметки на реальном списке целевых вузов (см. п.3) — от этого зависит, можно ли писать один универсальный парсер или потребуется адаптация под каждый вуз индивидуально.

## 7. Двухслойная модель данных вуза (итог)

Все данные с сайта вуза делятся на два качественно разных слоя:

**Слой 1 — обязательный, стандартизированный (закон, Приказ №1493).**
Адрес `/sveden/employees` + микроразметка `itemprop` — единые для всех вузов. Даёт «скелет»: ФИО, должность, степень, звание, дисциплины, стаж, факт работы в вузе. Это унифицированная точка входа и сверки, её можно масштабировать одним парсером. Контактов, возраста и даты защиты здесь по закону нет.

**Слой 2 — добровольный, индивидуальный у каждого вуза.**
Единого стандарта нет, под каждый вуз (иногда факультет) нужен свой парсер. Именно здесь лежат «жирные» поля — контакты и возраст. Проверено на двух вузах:

| Под-источник Слоя 2 | Что добавляет сверх обязательного | Пример |
|---|---|---|
| Страницы кафедр / поддомены | e-mail, телефон, кабинет, адрес | УрФУ: `linguistics-urgi.urfu.ru/.../staff/persons` |
| Индивидуальные страницы | год окончания, стаж «с года», научные интересы | УрФУ |
| Научные порталы (Pure и аналоги) | e-mail, кафедра, публикации | НГУ: `pure.nsu.ru` (email вида `i.surname@g.nsu.ru`) |
| Сайты диссертационных советов | дата защиты, тема, **год рождения**, научрук, PDF диссертации | НГУ: `dissovet.nsu.ru`; УрФУ: `dissovet.urfu.ru` |

Дополнительно по НГУ: в самом `/sveden/employees` есть встроенный поиск по ФИО/программе, единая таблица всех ППС (а не только разбивка по программам) и поле типа занятости (основное место / совместительство — сигнал мобильности). У УрФУ `employees` слабее (структура по институтам, без единой таблицы), зато сильнее кафедры и диссоветы.

**Практический алгоритм по одному вузу:**

```
1. /sveden/employees   → скелет (ФИО, степень, должность, дисциплины, стаж)
2. кафедра / Pure       → e-mail, телефон, детали
3. dissovet.<домен>     → возраст, дата защиты, тема (молодые после защиты)
4. ВАК (is_pilot=false + is_pilot=true) → внешняя сверка степени
```

**Связь с ВАК.** У национальных/федеральных вузов (НГУ, УрФУ) есть право самостоятельного присуждения степеней — их защиты публикуются в отдельной ветке ВАК: вкладка «Самостоятельное присуждение степеней» (`/adverts-list/independent`), в API — параметр `is_pilot=true` (~20 тыс. записей) vs `is_pilot=false` (~164 тыс.). Поэтому выпускника такого вуза нужно искать в обеих ветках, иначе есть риск «не найти», хотя запись существует.

**Короткая формулировка:** обязательная часть даёт единообразный скелет по всем вузам страны (ФИО, степень, дисциплины, факт работы). Добровольная часть даёт то, ради чего ищем — контакты и возраст, — но у каждого вуза по-своему, и её приходится собирать вручную под каждый вуз.
