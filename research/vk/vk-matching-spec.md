# Источник: ВКонтакте (контакт кандидата)

Спецификация поиска VK-профиля кандидата по данным из других источников (ВАК + сайт вуза).

**Статус:** черновик спеки для разработки
**Документация API:** [dev.vk.com](https://dev.vk.com/ru/reference), [VK ID](https://id.vk.com/about/business/go/docs/ru/vkid/latest/) (проверено 07.07.2026)

**MVP (этот этап):** поиск **только внутри паблика/группы вуза** (`users.search` + `group_id`). Город, возраст, широкий поиск, подписки, `groups.isMember` как отдельный скоринг — **в планах** (§9), не в объёме MVP.

---

## 1. Что хотим

Найти **один или несколько VK-профилей**, которые с высокой вероятностью принадлежат кандидату из карточки, и отдать проверяющему: ссылку, **скор** и расшифровку совпавших сигналов.

VK — **не самостоятельный источник кандидата**, а **дополнительный канал контакта**: без ФИО из ВАК/вуза искать некого. Мы **не выбираем единственного «победителя»** — если несколько профилей набрали высокий скор, отдаём всех выше порога, решение всегда за проверяющим.

**Когда запускается:** **обязательно и автоматически** на каждом полном прогоне пайплайна — пакетно по всем подходящим карточкам, **без** согласия оператора и без ручной кнопки. Не fallback «если нет email»: VK ищем всегда, параллельно/после слоя 2. См. [candidate-pipeline-architecture.md](../pipeline/candidate-pipeline-architecture.md), §7 и [app-architecture.md](../pipeline/app-architecture.md).

**Ограничение MVP:** без известного `university_vk_group_id` поиск по карточке **пропускаем** (статус `skipped_no_group`) — нечего сужать. Это ожидаемо для части `vak_no_site`, если `defend_org` не смапился на вуз из реестра с `vk_group_id`. Остальные карточки с группой обрабатываются всегда.

## 2. Зависимости от других источников (MVP)

| Поле | Обязательно на MVP? | Откуда | Как используем |
|---|---|---|---|
| Фамилия, Имя | **да** | ВАК / вуз | `users.search?q=` |
| `group_id` вуза | **да** | справочник `university_name → vk_group_id` (реестр вузов) | `users.search?group_id=` — **единственный** канал поиска на MVP |
| Отчество | нет | — | не используем |
| Пол | нет | ФИО / карточка | опционально `sex=` в том же запросе |
| Город, возраст, подписки, `universities` в профиле | нет на MVP | — | **в планах** (v2+) |

**Минимум MVP:** `{last_name, first_name, university_vk_group_id}`.

Справочник `vk_group_id` ведём в том же `university_registry`, что и домены сайтов — см. пайплайн §3.

## 3. Доступ к API

### 3.1. Получение токена

Через VK ID (`id.vk.com/about/business/go`), протокол OAuth 2.1 + PKCE:

1. Создать приложение в кабинете VK ID.
2. Сгенерировать `code_verifier`/`code_challenge` (S256), провести пользователя через `id.vk.com/authorize` с `PKCE`.
3. Обменять `code` на пару `access_token` + `refresh_token` через `POST id.vk.ru/oauth2/auth`.

**Токены:** `access_token` живёт **1 час**, `refresh_token` — **180 дней**. Автообновление — через `refresh_token` back-end запросом.

**Верификация бизнес-профиля (VK Бизнес ID):** с 2026 года приложения без подтверждённого бизнес-профиля блокируются — есть **60 дней отсрочки** с момента создания первого приложения. Для MVP не блокер; для продакшена — отдельная нетехническая задача заранее.

### 3.2. Почему не парсинг вместо API

- без логина `vk.com/search` отдаёт только первую страницу, дальше нужна авторизованная сессия;
- автоматический логин по паролю для сторонних приложений отключён;
- сайт — JS SPA → headless дороже официального JSON API;
- антибот агрессивнее к скрейпингу, чем к API.

### 3.3. Стоимость

API ВК бесплатное. Ограничения — технические (см. §8).

## 4. API-методы

### 4.1. Формат запроса

```
POST https://api.vk.ru/method/<method>
Authorization: Bearer <ACCESS_TOKEN>
Content-Type: application/x-www-form-urlencoded

v=5.199 + параметры метода
```

### 4.2. Какие методы используем

**MVP:**

| Метод | Токен | Назначение |
|---|---|---|
| `users.search` | пользователь | поиск по ФИ **внутри** `group_id` вуза |
| `users.get` | любой | детали профиля для простого скоринга; до 1000 `user_ids` за один вызов — обогащение всего пула кандидатов не по одному |
| `execute` | пользователь | обёртка до 25 вложенных вызовов (в т.ч. `users.search`) в один HTTP-запрос — см. §5.1, ключевой инструмент для пропускной способности на масштабе |
| `utils.resolveScreenName` / `groups.search` | любой/пользователь | один раз заполнить справочник `vk_group_id` вузов |

**В планах (v2+):** `groups.isMember`, `users.getSubscriptions`, `database.getCities`, расширенный каскад поиска.

Источники: [users.search](https://dev.vk.com/ru/method/users.search).

## 5. MVP: поиск только в группе вуза

Один запрос (без каскада город → возраст → весь VK):

```
POST https://api.vk.ru/method/users.search
  ?q=Иванов+Алексей
  &group_id=<vk_group_id вуза>
  &sex=2                      ← опционально, если известен
  &count=200
  &fields=domain,screen_name,city,bdate,universities,career,occupation,status
  &v=5.199
```

Если `group_id` неизвестен → сразу `skipped_no_group`, без вызова API.

Ограничения `users.search`: только пользовательский токен; профили со скрытым поиском не возвращаются; участники паблика со скрытым списком участников могут не попасть в выдачу с `group_id` — на MVP это принимаем как `not_found`, расширенный fallback — в планах.

**Как один раз получить `group_id` вуза (справочник, не на каждый поиск):**

- `utils.resolveScreenName?screen_name=urfu` или
- `groups.search?q=УрФУ&type=page`

### 5.1. Батчинг через `execute` — почему это не «3 запроса в секунду»

Формальный лимит VK API на пользовательский токен — **3 запроса/сек** (подтверждено на `dev.vk.com/en/api/api-requests` и в официальном FAQ VK). Но это лимит на **HTTP-вызов**, а не на «единицу работы»: метод `execute` пакует до **25** вложенных вызовов произвольных методов (в т.ч. `users.search`) в один HTTP-запрос, и сам `execute` считается как один вызов в рамках тех же 3/сек. Эффективная пропускная способность — не 3, а **до ~75 `users.search` в секунду на один токен** (3 × 25).

Пример код-обёртки (`execute`, VKScript):

```
var i = 0;
var result = [];
while (i < 25 && i < candidates.length) {
  result.push(API.users.search({
    "q": candidates[i].last_name + " " + candidates[i].first_name,
    "group_id": candidates[i].group_id,
    "count": 200,
    "v": "5.199"
  }));
  i = i + 1;
}
return result;
```

**На порядок величины страны (~100 000 карточек):** 100 000 / 75 ≈ **22 минуты** теоретически — против часов при наивном последовательном вызове `users.search` по одному. Это радикально меняет оценку «сколько времени займёт VK-шаг на полном прогоне» (см. `app-architecture.md`, §8).

**Важная оговорка.** Расчёт — теоретический потолок по официальному лимиту. Есть отдельный недокументированный «мягкий» антифлуд-порог самого `users.search` (§8.1 ниже, ~25–40 вызовов за короткое окно, после которого VK молча отдаёт пустой результат вместо ошибки) — неизвестно, считает ли этот механизм вызовы внутри `execute` так же, как обычные отдельные вызовы. **Нужно проверить эмпирически** на пилотном батче (несколько тысяч карточек) до того, как закладывать 22 минуты как рабочую цифру, а не только формальный расчёт по документированному лимиту.

## 6. Алгоритм MVP

### Фаза 0. Подготовка входа

```
Обязательно: last_name, first_name, university_vk_group_id
Опционально: sex
```

Карточки `site_and_vak` / `site_and_vak_probable` / `site_no_vak` — вуз уже есть → резолвим `vk_group_id` из реестра.

`vak_no_site` — только если `defend_org` смапился на вуз с `vk_group_id`; иначе `skipped_no_group`.

Отчество в `q` не передаём.

### Фаза 1. Сбор пула

```
if not university_vk_group_id:
    return skipped_no_group

pool = users.search(q=ФИ, group_id=..., sex?=...)
if pool empty → not_found
else → Фаза 2
```

### Фаза 2. Обогащение (лёгкое)

`users.get` батчем по id из пула — поля для скоринга имени/вуза в профиле (без подписок и без `groups.isMember` на MVP).

### Фаза 3. Скоринг MVP (упрощённый)

| Сигнал | Баллы |
|---|---|
| Фамилия + имя совпали | +30 (обязательный минимум для пула) |
| Вуз в `universities` (fuzzy) | +20 |
| Вуз/«преподаватель» в `career`/`occupation`/`status` | +10 |
| Пол в профиле не соответствует ожидаемому | −15 |

Факт «нашёлся через поиск в группе вуза» уже заложен входом (`group_id`) — отдельный бонус за членство на MVP не дублируем.

**Пороги (черновые, калибровка позже):** ≥40 `high`, 25–39 `medium`, ниже — не отдаём (или отдаём все из маленького пула с пометкой `low`, если пул ≤3 — на усмотрение UI). Решение всегда за проверяющим.

### Фаза 4. Ссылка

`https://vk.com/{screen_name}` или `https://vk.com/id{user_id}`.

```json
{
  "input": {
    "last_name": "Иванов",
    "first_name": "Алексей",
    "university": "УрФУ",
    "university_vk_group_id": 12345
  },
  "status": "candidates_found",

      "profile_url": "https://vk.com/id123456789",

      "confidence": "high",
```

## 8. Ограничения и риски

### 8.1. Технические лимиты

| Ограничение | Значение |
|---|---|
| Rate limit (документированный, подтверждено на `dev.vk.com`) | **3 запроса/сек** на пользовательский токен (20/сек — токен сообщества, но `users.search` требует именно пользовательский, см. §5) |
| Эффективная пропускная способность через `execute` | до **~75 `users.search`/сек** на токен (25 вложенных вызовов × 3 execute/сек) — см. §5.1. Это не отдельный лимит VK, а следствие батчинга в рамках официального лимита |
| `users.search` — макс. результатов | **1000** результатов на один вызов с данным набором параметров — это лимит выдачи одного запроса, **не** лимит запросов в секунду (легко спутать: «1000» и «3/сек» — про разные вещи) |
| **Скрытый лимит** `users.search` | ~25–40 вызовов за короткое окно → молча `count:0, items:[]`. Не подтверждён документацией VK, только наблюдениями сообщества разработчиков; нужен sanity-check на реальном токене + backoff. Неизвестно, применяется ли этот порог к вызовам внутри `execute` так же, как к обычным — проверять эмпирически |
| Коды ошибок | 5 (токен → refresh), 6/9 (флуд → backoff), 15, 18, 29, 30 |

### 8.2. Качество данных (MVP)

| Риск | Митигация |
|---|---|
| Нет `vk_group_id` у вуза | `skipped_no_group`; дозаполнить справочник |
| Кандидат не в паблике вуза / скрыт из поиска | `not_found` на MVP; широкий поиск — в планах |
| Однофамильцы в одной группе | скоринг + ручная проверка |
| Иностранные ФИО (транслит) | ожидаемый `not_found` |

## 9. Порядок внедрения

**MVP (сейчас):**
1. Справочник `university → vk_group_id` (часть `university_registry`).
2. `users.search` **только** с `group_id` + ФИ (+ опционально `sex`).
3. Лёгкий скоринг (имя / вуз в профиле) → выдача списка проверяющему.
4. Токен вручную через VK ID (60 дней до верификации — не блокер).

**В планах (не MVP):**

| Этап | Что добавляем |
|---|---|
| v2 | Каскад: город (`city`) → возраст → широкий `users.search` без группы; `groups.isMember`; подписки `getSubscriptions`; `maiden_name` / `career` в скоринге; авто-refresh токена |
| v3 | Калибровка весов на размеченной выборке; мониторинг выбора проверяющего; `execute`-батчинг; верификация бизнес-профиля VK ID |

Каскад «от узкого к широкому» (бывшие шаги C → B → D → A) и полный скоринг с городом/возрастом/подписками остаются спецификацией **будущего** этапа — при реализации v2 вернуть развёрнутое описание каскада из истории документа / дописать сюда заново.
