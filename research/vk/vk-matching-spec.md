# Источник: ВКонтакте (контакт кандидата)

Спецификация поиска VK-профиля кандидата по данным из других источников (ВАК + сайт вуза).

**Статус:** черновик спеки для разработки
**Документация API:** [dev.vk.com](https://dev.vk.com/ru/reference), [VK ID](https://id.vk.com/about/business/go/docs/ru/vkid/latest/) (проверено 07.07.2026)

---

## 1. Что хотим

Найти **один или несколько VK-профилей**, которые с высокой вероятностью принадлежат кандидату из карточки, и отдать проверяющему: ссылку, **скор (0–100)** и уровень уверенности, расшифровку совпавших сигналов.

VK — **не самостоятельный источник кандидата**, а **дополнительный канал контакта**: без ФИО из ВАК/вуза искать некого. Мы **не выбираем единственного «победителя»** — если несколько профилей набрали высокий скор, отдаём всех выше порога, решение всегда за проверяющим.

**Когда запускается:** вручную, по кнопке на карточке конкретного кандидата (не пакетно по всей базе) — см. [candidate-pipeline-architecture.md](../pipeline/candidate-pipeline-architecture.md), §7. Обычно — когда слой контактов с сайта вуза не дал email/телефон.

## 2. Зависимости от других источников

| Поле | Обязательно? | Откуда | Как используем |
|---|---|---|---|
| Фамилия, Имя | **да** | ВАК / вуз | `users.search?q=` |
| Отчество | нет | — | **не используем** (в VK не отображается и нет отдельного поля в API) |
| Пол | нет | ФИО / карточка | `sex=1|2` — почти бесплатно режет шум вдвое |
| Город | нет | вуз / город защиты | фильтр `city` + скоринг |
| Вуз | нет | вуз / ВАК | скоринг по `universities`, `groups.isMember`, подпискам |
| Возраст (оценка) | нет | дата защиты, стаж | `age_from/age_to` + скоринг по `bdate` |
| Дата защиты (точная) | нет | ВАК (через мэтч в объединённой карточке) | точнее оценивает возраст, чем формула по стажу — см. Фазу 0 |
| `maiden_name` | нет | — | скоринг для женщин |
| `group_id` вуза | нет | справочник (once) | доп. поиск + `groups.isMember` |

**Минимум:** `{last_name, first_name}`. **Расширенный вход:** + `sex`, `city`, `university_name`, `age_range`, `university_vk_group_id`.

## 3. Доступ к API

### 3.1. Получение токена

Через VK ID (`id.vk.com/about/business/go`), протокол OAuth 2.1 + PKCE:

1. Создать приложение в кабинете VK ID.
2. Сгенерировать `code_verifier`/`code_challenge` (S256), провести пользователя через `id.vk.com/authorize` с `PKCE`.
3. Обменять `code` на пару `access_token` + **`refresh_token`** через `POST id.vk.ru/oauth2/auth`.

**Токены:** `access_token` живёт **1 час**, `refresh_token` — **180 дней**. Автообновление — через `refresh_token` back-end запросом, без ручной переавторизации каждый час.

**Верификация бизнес-профиля (VK Бизнес ID):** с 2026 года приложения без подтверждённого бизнес-профиля блокируются — но есть **60 дней отсрочки** с момента создания первого приложения. Для MVP это не блокер (можно стартовать сразу), но для продакшена верификацию нужно закладывать в план как отдельную (нетехническую) задачу заранее, а не в последний момент.

### 3.2. Почему не парсинг вместо API

- без логина `vk.com/search` отдаёт только первую страницу выдачи, дальше требует авторизованную сессию — та же проблема с доступом, что и у API, только решается более хрупко;
- автоматический логин по паролю ВК отключил для сторонних приложений (работает только с подстановкой `client_id` официальных клиентов ВК — это маскировка под чужое приложение, грубое нарушение правил);
- сайт — JS SPA, нужен headless-браузер, а не HTTP+парсинг — на порядок больше инфраструктуры, чем вызов версионированного JSON API;
- антибот-защита реагирует на скрейпинг агрессивнее, чем на официальный API-трафик.

### 3.3. Стоимость

API ВК бесплатное. Ограничения — не денежные, а технические (см. §6).

## 4. API-методы

### 4.1. Формат запроса

```
POST https://api.vk.ru/method/<method>
Authorization: Bearer <ACCESS_TOKEN>
Content-Type: application/x-www-form-urlencoded

v=5.199 + параметры метода
```

### 4.2. Какие методы используем

| Метод | Токен | Назначение |
|---|---|---|
| `users.search` | пользователь | сбор пула кандидатов по ФИ |
| `users.get` | любой | детали профиля для скоринга (батч до 1000 id) |
| `users.getSubscriptions` | пользователь/сервисный | подписки (батч не поддерживается — 1 запрос на профиль) |
| `groups.isMember` | любой | членство в паблике вуза (батч до 500 id) |
| `groups.getById`, `groups.search` | любой/пользователь | метаданные / поиск `group_id` вуза |
| `database.getCities` | любой | `city_id` по названию (кэш) |
| `utils.resolveScreenName` | любой | `group_id` по короткому имени |

Источники: [users.search](https://dev.vk.com/ru/method/users.search), [users.getSubscriptions](https://dev.vk.com/ru/method/users.getSubscriptions), [groups.isMember](https://dev.vk.com/ru/method/groups.isMember).

## 5. Основной поиск (каскад от узкого к широкому)

**Принцип:** после каждого шага — дешёвый скоринг; при первом кандидате со скором ≥55 **останавливаемся**, следующие шаги не выполняем. Экономит запросы и бережёт лимиты.

**Базовый запрос** (общий для всех шагов):

```
POST https://api.vk.ru/method/users.search
  ?q=Иванов+Алексей
  &sex=2                      ← если известен, из ФИО/карточки
  &count=200
  &fields=domain,screen_name,city,bdate,maiden_name,universities,career,occupation,status,home_town
  &v=5.199
```

**Доп. параметры по шагам** (пропускаем шаг, если нет входных данных):

| Шаг | Доп. параметры | Условие запуска | Узость | Стоп |
|---|---|---|---|---|
| C | `group_id=<vk_group_id вуза>` | известен `group_id` | очень узкий | скор ≥55 |
| B | `city=<id>` | известен город | средний | скор ≥55 |
| D | `city=<id>&age_from=&age_to=` | город + возраст | узкий по фильтрам, шум по возрасту (±4 года) | скор ≥55 |
| A | (без фильтров, fallback) | всегда | широкий | `count≈1000` → `ambiguous` |

Результаты шагов объединяем по `id` (дедуп). Ограничения `users.search`: только пользовательский токен, максимум **1000** результатов даже с `offset`, профили со скрытым поиском не возвращаются.

**Пример lookup для доп. параметров:**
- `database.getCities?q=Екатеринбург&country_id=1` → `city_id` (лимит `q` — 15 символов);
- `utils.resolveScreenName?screen_name=urfu` или `groups.search?q=УрФУ&type=page` → `group_id`.

## 6. Алгоритм

### Фаза 0. Подготовка входа

```
Обязательно: last_name, first_name
Опционально: sex, city → city_id, university → group_id,
             defense_year/experience → estimated_age_range (± 4 года)
```

Если карточка пришла из объединённого пайплайна (`match_status = confirmed`/`likely`, см. [candidate-pipeline-architecture.md](../pipeline/candidate-pipeline-architecture.md)) — `university` и точная `defense_date` уже готовы, не нужно резолвить их заново или оценивать возраст по стажу: `defense_date` даёт более узкий и точный `estimated_age_range`, чем общая формула. Формула «по стажу/дате защиты» из этого раздела — fallback для карточек без такого мэтча (`vak_only`/`site_only`).

Отчество в `q` не передаём.

### Фаза 1. Сбор пула (каскад §5, с ранним выходом)

```
pool = {}
for step in [C, B, D, A]:        # пропуск, если нет входных данных
    if not step.applicable: continue
    pool = dedupe(pool ∪ step.results)
    score_cheap(pool)
    if any(p.score >= 55 for p in pool): break
```

После каскада: пул пуст → `not_found`; последний шаг A и `count≈1000` → `ambiguous` (подписки не тянем); иначе → Фаза 2.

### Фаза 2. Обогащение (только для финального пула, не после каждого шага)

1. `users.get` — полный профиль (батч).
2. `groups.isMember` — членство в паблике вуза (батч по 500).
3. `users.getSubscriptions` — подписки открытых профилей (по 200/стр., 1 запрос на профиль); матчим названия по словарю вуза/города.

### Фаза 3. Скоринг (гибкий, отсутствие данных = 0, не штраф)

| Сигнал | Баллы |
|---|---|
| Фамилия + имя совпали | +30 (обязательный минимум для пула) |
| `maiden_name` совпала (для женщин) | +25 |
| Вуз в `universities` (fuzzy) | +20 |
| **Связан с пабликом вуза** (см. ниже, один раз) | +25 |
| Возраст в диапазоне ±4 года | +15 |
| Возраст явно вне диапазона | −10 |
| Город профиля = целевому | +15 / другой −5 / не указан 0 |
| Город в подписках | +10 |
| Вуз/«преподаватель» в `career`/`occupation`/`status` | +10 |
| Пол в профиле не соответствует ожидаемому | −15 |

«Связан с пабликом вуза» — один факт, детектируемый тремя способами (найден через шаг C / `groups.isMember=1` / вуз в подписках), считается **один раз**, независимо от числа сработавших детекторов.

**Пороги:** ≥55 `high`, 35–54 `medium`, <35 → проверяем **динамический порог**: `threshold = (max_score + avg_score) / 2`. Отдаём профили `≥ threshold` как `relative`, только если `max−min ≥ 5` и `max ≥ 20` (иначе разброса/сигнала нет — не отдаём).

*Пример:* пул из 10 однофамильцев, двое по 30, восемь по 5 → `threshold = (30+10)/2 = 20` → отдаём двоих с 30 как `relative`.

### Фаза 4. Ссылка

`https://vk.com/{screen_name}` или `https://vk.com/id{user_id}`.

## 7. Формат ответа

```json
{
  "input": { "last_name": "Иванов", "first_name": "Алексей", "city": "Екатеринбург", "university": "УрФУ" },
  "status": "candidates_found",
  "vk_candidates": [
    {
      "vk_id": 123456789,
      "profile_url": "https://vk.com/id123456789",
      "score": 72,
      "confidence": "high",
      "matched_signals": ["name", "university_education", "university_group_signal (via subscription)", "age_in_range", "city"]
    }
  ],
  "meta": { "pool_size": 47, "api_requests_made": 6, "searched_at": "2026-07-07T12:00:00+05:00" }
}
```

**Статусы:** `candidates_found` (есть `high`/`medium`) · `candidates_found_relative` (сработал динамический порог) · `ambiguous` (широкий пул, лидера нет) · `not_found` · `partial` (частичное обогащение) · `error`.

**Проверяющий видит:** ссылка + скор + уровень + какие сигналы совпали — сам открывает профиль и решает.

## 8. Ограничения и риски

### 8.1. Технические лимиты

| Ограничение | Значение |
|---|---|
| Rate limit (документированный) | 3 запроса/сек |
| **Скрытый лимит `users.search`** (недокументированный) | ~25–40 вызовов за короткое окно → временная блокировка, метод **молча отдаёт `count:0, items:[]`** (не ошибку!). Нужен sanity-check контрольным частым именем + backoff, прежде чем ставить `not_found` — иначе легко принять рейт-лимит за «профиля нет» |
| `users.search` — макс. результатов | 1000 на набор параметров |
| `users.getSubscriptions` — count | 200/запрос, батч не поддерживается |
| `groups.isMember` — user_ids | 500/запрос |
| Коды ошибок для обработки | 5 (токен истёк → refresh), 6/9 (флуд-контроль → backoff), 15 (доступ запрещён), 18 (страница удалена/забанена → исключить из пула), 29 (лимит метода), 30 (приватный профиль → сигнал 0) |

Рекомендация: `execute` для батчинга (до 25 вложенных вызовов) — особенно для `users.getSubscriptions`, которая не батчуется нативно.

### 8.2. Качество данных

| Риск | Митигация |
|---|---|
| Профиль скрыт из поиска / закрыт | `not_found` / скоринг без подписок |
| Однофамильцы | скоринг + ручная проверка |
| Иностранные ФИО (транслитерация, напр. кит. фамилии из ВАК) | поиск по кириллице систематически не найдёт — ожидаемый `not_found`, не дефект алгоритма |
| Старый/заброшенный профиль | проверяющий видит `last_seen`, если доступен |
| Участники паблика вуза скрыты | fallback на подписки кандидата |
| Веса и пороги скоринга не откалиброваны на реальных данных | см. §9 — калибровка через размеченную выборку до продакшена |

## 9. Порядок внедрения

**MVP:** справочник городов → `users.search` по ФИ (+город, +sex) → скоринг (имя/город/universities/возраст) → выдача списка. Токен получаем вручную через VK ID (60 дней до требования верификации — не блокер для старта).

**v2:** справочник `group_id` вузов → `groups.isMember` + `getSubscriptions` → `maiden_name`, `career` → авто-refresh токена.

**v3:** калибровка весов/порогов на размеченной выборке (кандидаты с уже известным верным VK-профилем — считаем precision/recall) → мониторинг: логирование результатов + сбор выбора проверяющего для дообучения весов → кэш справочников, `execute`-батчинг, верификация бизнес-профиля VK ID.

## 10. Связь с другими источниками

```
ВАК ──────────┐
              ├─► Матчер ─► Карточка кандидата ─► Слой 2 (контакты)
Сайт вуза ────┘                                        │
                                          контакт не найден / нужен доп. канал
                                                        │
                                                        ▼
                                          Проверяющий жмёт «Искать в VK»
                                                        │
                                                        ▼
                                    VK Matching (ФИ + университет + возраст из даты защиты)
                                                        │
                                                        ▼
                                    Список VK-профилей + скор → проверяющий выбирает
```

Полная схема сборки карточки (матчер ВАК/вуз, контракт со слоем контактов) — в [candidate-pipeline-architecture.md](../pipeline/candidate-pipeline-architecture.md).

**Коротко:** VK даёт не «истину о кандидате», а список вероятных профилей с оценкой. Решение всегда за человеком.
